Check-expiry and Slack notification workflow
============================================

Purpose
-------
A reusable GitHub Actions workflow that reports the Snyk vulnerabilities tracked
in an environment and how close each is to becoming non-compliant. It sends a
Slack message to the cyber-dojo-alerts channel and writes a GitHub step summary.
It makes NO Kosli calls and has no effect on compliance status.

It is called once per environment by aws-beta.yml and aws-prod.yml, after that
environment's scan has run. The per-vuln data it reports on is produced earlier
in the same GitHub Actions run by artifact_snyk_test.yml, which uploads one
vuln-*.json file per vulnerability.


Two expiry mechanisms
---------------------
Each vuln-*.json file is the per-vuln matrix data from artifact_snyk_test.yml,
with first_seen_ts (the trail's created_at) added. find_expiring_vulns.py reads
these fields:

  first_seen_ts          unix timestamp the per-vuln trail was first created
  severity               "critical" | "high" | "medium" | "low"
  ignore_expires_exists  true if the .snyk file has an ignore entry for this vuln
  ignore_forever         true if that ignore entry has no expiry date
  ignore_expires_ts      unix timestamp of that ignore entry's expiry (0 if none)
  ignore_expires         string form of that expiry (empty if none)
  full_id                e.g. "SNYK-ALPINE322-ZLIB-16078399"
  vuln_url               e.g. "https://security.snyk.io/vuln/SNYK-..."
  trail_name             e.g. "creator-medium-SNYK-ALPINE322-ZLIB-16078399"

The rego policy (snyk-vuln-compliance.rego) uses max_days_by_severity from
rego.params.{env}.json:

  aws-beta:  critical=1, high=2, medium=4, low=10
  aws-prod:  critical=0, high=2, medium=4, low=10

Each tracked vuln is reported under exactly one of two mechanisms:

  dot_snyk_expiry -- the vuln has a .snyk ignore entry with an expiry date.
    days_remaining = (ignore_expires_ts - now_ts) / 86400
    (A .snyk ignore entry with no expiry date is "ignore forever": compliant
    regardless of age and never reported.)

  rego_limit -- the vuln has no .snyk ignore entry, so the rego age limit applies.
    age_days = (now_ts - first_seen_ts) / 86400
    days_remaining = max_days_by_severity[severity] - age_days

days_remaining is positive while the vuln is still compliant and zero or negative
once it is already non-compliant. Both compliant and already-non-compliant vulns
are reported (the report is informational; the nightly scan owns the verdict).


.github/workflows/check-expiry-and-notify.yml
----------------------------------------------
Trigger: workflow_call
  inputs:  kosli_env (required)
  secrets: SLACK_WEBHOOK_URL (required)

env block:
  GITHUB_RUN_URL     link to this Actions run
  RESPONSE_GUIDE_URL link to docs/expiry-response.md

Single job: check-expiry (runs-on: ubuntu-latest)
  1. actions/checkout@v6
  2. actions/download-artifact@v8 -- pattern vuln-*.json, merge-multiple, into vulns/
  3. Find vulns:
       ./bin/find_expiring_vulns.py --env <kosli_env> --vuln-dir vulns
       Then jq builds, into GITHUB_ENV:
         VULNS_JSON    the .vulns array
         STATUS_SYMBOL :x: if there are vulns and the soonest is already
                       non-compliant (days_remaining <= 0), else :white_check_mark:
         SLACK_MESSAGE "No Snyk vulnerabilities :-)" when the list is empty,
                       otherwise the soonest vuln's days line ("Days till
                       non-compliant: N" or "Days non-compliant: N"), full_id,
                       severity, repo (artifact), and mechanism.
  4. Send Slack:
       "Environment: <kosli_env> <STATUS_SYMBOL>\n<SLACK_MESSAGE>\n
        See: <GITHUB_RUN_URL>\nResponse guide: <RESPONSE_GUIDE_URL>"
  5. Send Slack -- workflow failure (if: failure())
  6. Write GitHub step summary:
       ./bin/print_expiring_vulns_summary.py --env <kosli_env> --vulns "$VULNS_JSON"
       >> $GITHUB_STEP_SUMMARY


.github/actions/send-slack-message/action.yml
----------------------------------------------
Composite action. Kept local so snyk-scanning has no cross-repo action dependency.

  inputs:
    message:   required
    slack_url: required

  Posts a single Slack "section" block (mrkdwn) containing the message via
  curl -X POST --fail to slack_url.


bin/find_expiring_vulns.py
--------------------------
CLI:
  ./bin/find_expiring_vulns.py --env <env> --vuln-dir <dir>

Reads rego.params.{env}.json for the severity limits, globs vuln-*.json in
<dir>, and for each file emits at most one result (dot_snyk_result OR
rego_result, depending on whether a .snyk ignore entry exists). Writes a JSON
object to stdout and exits 0:

  {
    "vulns": [
      {
        "env":            "aws-beta",
        "trail_name":     "runner-high-SNYK-GOLANG-...",
        "full_id":        "SNYK-GOLANG-...",
        "severity":       "high",
        "vuln_url":       "https://security.snyk.io/vuln/...",
        "mechanism":      "dot_snyk_expiry" | "rego_limit",
        "days_remaining": 19.8,
        "ignore_expires": "2026-06-01 10:53:10+00:00" | null,
        "age_days":       5.16 | null,
        "limit_days":     10 | null,
        "artifact":       "runner"
      }
    ]
  }

The list is sorted by days_remaining ascending, so the soonest-to-expire (or
most-overdue) vuln is first. "artifact" is the trail_name segment before the
first -severity- part. now_ts is the script's own time.time() (not any timestamp
from the attestation).


bin/print_expiring_vulns_summary.py
-----------------------------------
CLI:
  ./bin/print_expiring_vulns_summary.py --env <env> --vulns '<json array>'

The --vulns value is the .vulns array from find_expiring_vulns.py. Writes
Markdown to stdout for the GitHub step summary:

  ## <env> (Snyk vulns tracked: Count=N)

  then one table per severity, in order critical, high, medium, low:

  ### <Severity> (Count=N)

  | Artifact | Days remaining | Mechanism | Vuln ID |

  (or "No vulnerabilities." when that severity has none). Rows are sorted by
  days_remaining ascending. Mechanism is displayed as "rego" (rego_limit) or
  ".snyk" (dot_snyk_expiry). Vuln ID is a Markdown link [full_id](vuln_url).

Example output:

  ## aws-beta (Snyk vulns tracked: Count=2)

  ### Critical (Count=0)

  No vulnerabilities.

  ### High (Count=1)

  | Artifact | Days remaining | Mechanism | Vuln ID |
  |----------|----------------|-----------|---------|
  | runner | 20 | .snyk | [SNYK-GOLANG-GOLANGORGXNETHTTP2-16535157](https://security.snyk.io/vuln/SNYK-GOLANG-GOLANGORGXNETHTTP2-16535157) |

  ### Medium (Count=0)

  No vulnerabilities.

  ### Low (Count=1)

  | Artifact | Days remaining | Mechanism | Vuln ID |
  |----------|----------------|-----------|---------|
  | creator | 5 | rego | [SNYK-ALPINE322-NGHTTP2-16426989](https://security.snyk.io/vuln/SNYK-ALPINE322-NGHTTP2-16426989) |


What this workflow does NOT do
------------------------------
- It does not run Snyk scans.
- It does not create or modify any Kosli trails, attestations, or flows.
- It makes no Kosli calls at all; it reads the run's local vuln-*.json files.
- It does not affect the compliance status of any artifact or environment.


docs/expiry-response.md
-----------------------
Linked from the Slack message. Tells the reader what to do when a vuln is
approaching (or past) its expiry, depending on which mechanism applies.
